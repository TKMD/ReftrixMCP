### 前文

Reftrixプロジェクトは、AGPL-3.0ライセンスのオープンソース版と商用ライセンスのデュアルライセンス戦略を採用しています。このデュアルライセンス体制を維持しながら、プロジェクトの長期的な持続可能性を確保するために、本貢献者ライセンス契約（CLA）が必要です。

本CLAは以下の目的を持ちます:

1. **貢献者の権利保護**: 貢献者は自身の著作権を保持し続けます。本契約は著作権の譲渡ではなく、ライセンスの付与です。
2. **プロジェクトの持続可能性**: デュアルライセンス体制により、オープンソースコミュニティと商用ユーザーの双方を支援できます。
3. **法的明確性**: 貢献に関する権利と義務を明確にすることで、将来の法的紛争を防止します。

本CLAは、Apache Software Foundation の Individual Contributor License Agreement（ICLA）、および MongoDB、GitLab、HashiCorp 等のデュアルライセンスプロジェクトのベストプラクティスを参考に作成されています。

### 第1条: 定義

本契約において、以下の用語は次の意味を持ちます:

**「貢献（Contribution）」**
あなたがプロジェクトに対して提出するすべての著作物を指します。これには、ソースコード、ドキュメント、テスト、設定ファイル、その他のコンテンツが含まれますが、これらに限定されません。GitHub Pull Request、Issues内のコード、メール、その他の手段で提出されたものを含みます。

**「貢献者（Contributor）」**
本契約に合意し、プロジェクトに貢献するすべての個人または法人を指します。法人の場合、その法人によって管理されるすべての関連エンティティも含まれます。

**「プロジェクト（Project）」**
Reftrix（https://github.com/TKMD/ReftrixMCP）およびその関連するすべてのリポジトリ、コードベース、成果物を指します。

**「プロジェクト管理者（Maintainer）」**
プロジェクトの管理・運営を行う主体（現時点ではReftrix開発チーム）を指します。

### 第2条: 著作権ライセンスの付与

本契約に同意することにより、あなたは現在および将来のプロジェクト管理者に対して、以下の条件でライセンスを付与します:

**付与するライセンスの内容:**

- **永続的（Perpetual）**: 期間の制限なく有効
- **取消不能（Irrevocable）**: 一度付与したライセンスは取り消せない
- **世界的（Worldwide）**: 地域的制限なし
- **非独占的（Non-exclusive）**: あなたも同一の権利を保持する
- **ロイヤリティフリー（Royalty-free）**: 使用料・対価の支払いなし
- **再ライセンス権を含む（Sublicensable）**: プロジェクト管理者は第三者に再ライセンス可能

**ライセンスの対象となる行為:**
- 複製（Reproduction）
- 派生著作物の作成（Derivative Works）
- 公衆送信・配布（Public Communication and Distribution）
- 実演・展示（Performance and Display）
- 上記の行為を第三者に許可すること（再ライセンス）

**重要: 著作権の帰属**

あなたは自身の貢献に関する著作権を引き続き保持します。本条項は、あなたからプロジェクト管理者への著作権の譲渡ではなく、ライセンスの付与です。

**再ライセンスに関する注記**

再ライセンス権の付与は、Reftrixのデュアルライセンス戦略（AGPL-3.0オープンソース版と商用ライセンス版）を維持するために不可欠です。これにより、プロジェクト管理者は以下が可能になります:

- あなたの貢献をAGPL-3.0ライセンスのオープンソース版に含める
- あなたの貢献を商用ライセンス版に含める
- 将来的にライセンスモデルを変更する（オープンソース性は維持される）

**特許による攻撃的行動の禁止**

### 第4条: 表明と保証

本契約に同意することにより、あなたは以下を表明し保証します:

**4.1 オリジナル著作物**

あなたの貢献は、あなた自身のオリジナル著作物であり、以下のいずれかの条件を満たします:
- 貢献全体があなた自身が創作したもの
- 第三者のオープンソースライセンス（MIT、Apache-2.0、BSD-3-Clause等）の著作物を含む場合、その旨を明示している

**4.2 法的権限**

あなたは、本契約を締結し、本契約に定めるライセンスを付与するための完全な法的権限を有しています。

**4.3 雇用関係・委託関係**

あなたの貢献が職務上の著作物に該当する可能性がある場合（雇用中や業務委託中に創作した場合等）、あなたは雇用主または委託元から、その貢献をプロジェクトに提出するための明示的な許可を得ています。

**4.4 第三者の権利**

あなたの貢献は、知的財産権（著作権、特許権、商標権等）を含む第三者の権利を侵害しません。

**4.5 告知義務**

あなたの貢献に、第三者が著作権またはその他の権利を主張している可能性があることをあなたが知っている場合、あなたはプロジェクト管理者に対してその旨を速やかに通知します。

あなたの雇用主が、あなたが勤務中に創作した著作物に対する権利を主張する場合、あなたの雇用主も本CLAに署名するか、または本CLAへの同意を書面で提出してください。貢献を法人名義で提出する場合、その法人の権限を有する代表者が本CLAに同意する必要があります。

### 第7条: 署名方法

#### 方法1: GitHub Pull Requestでのコメント署名（推奨）

Pull Requestのコメントに以下の文言を投稿することで、本CLAに同意したものとみなします:

```
I have read the CLA Document and I hereby agree to its terms.
（本CLAを読み、その条件に同意します。）
```

このコメントは、Pull Request作成後にコメントとして投稿してください。

#### 方法2: 署名ファイルのコミット

以下のいずれかのパスに署名ファイルを作成・コミットすることでも署名できます:

**ファイルパス**: `.github/CLA-signatures/<GitHubユーザー名>.md`

**署名ファイルのテンプレート**:

```markdown
# CLA署名

- **名前（Name）**: [フルネーム]
- **GitHubユーザー名**: [GitHubユーザー名]
- **メールアドレス**: [メールアドレス]
- **所属（任意）**: [組織名または個人]
- **署名日**: [YYYY-MM-DD]

本CLAを読み、その条件に同意します。

I have read the CLA Document and I hereby agree to its terms.
```

### 第8条: DCO（Developer Certificate of Origin）との併用

本CLAと併せて、DCO（Developer Certificate of Origin）の使用を推奨します。コミットメッセージに以下のタグを含めることで、DCOの要件を満たします:

```
Signed-off-by: Your Name <your.email@example.com>
```

Gitコマンドを使用する場合:

```bash
git commit -s -m "feat(search): implement semantic search"
```

DCOは以下を証明します:
- あなたが著作権を持つ著作物を提出していること
- プロジェクトのライセンスで提出する権限があること
- 他のすべての証明の事実に同意していること

DCOの詳細は https://developercertificate.org/ を参照してください。

### 第10条: 改定

**10.1 改定の条件**

プロジェクト管理者は、以下のいずれかに該当する合理的な理由がある場合に限り、本CLAを改定することができます:

- (a) 適用法令の変更または新たな法令の施行に対応するため
- (b) 裁判所の判決または規制当局のガイダンスに従うため
- (c) プロジェクトのライセンス体制の変更に伴い必要となるため
- (d) セキュリティまたはプライバシー上の問題に対処するため
- (e) 業界標準またはベストプラクティスの変化に対応するため

**10.2 通知義務**

改定版CLAは、本リポジトリでの公開に加え、以下の方法で貢献者に通知されるものとします:

- (a) リポジトリのリリースノートまたはCHANGELOGへの記載
- (b) 改定日から少なくとも30日間の事前通知期間の設定

**10.3 既存の貢献への影響**

既に署名済みの貢献については、署名時点のCLAが引き続き適用されます。改定後も継続して貢献する場合は、改定版CLAへの同意が必要です。

**10.4 重要な変更に関する特則**

第2条（著作権ライセンスの付与）、第3条（特許ライセンスの付与）、第9条（準拠法・管轄）の実質的な変更を伴う改定については、貢献者からの明示的な再同意を求めるものとします。
